## The Pitch That Hooks You

You're evaluating frameworks for a business application. You need forms, workflows, role-based permissions, REST APIs, a database layer, background jobs, and an admin interface. Building all of that from scratch sounds like six months of work you don't have.

Then you find Frappe.

Full-stack. Low-code. Batteries-included. Define a "DocType" in the UI, and the framework generates your database table, REST API, form view, list view, and permission model — automatically. Need an ERP? ERPNext sits on top and gives you accounting, inventory, HR, and CRM out of the box. The GitHub stars are healthy. The documentation looks comprehensive. The demo is *slick*.

You spin up a bench, create your first DocType, watch the form materialize from thin air, and think: *this is the future of rapid application development*.

I thought the same thing. So did a lot of developers. What follows is what we found after the honeymoon ended — not opinion dressed as fact, but a catalogue of architectural decisions that the framework's own maintainers have acknowledged as problematic, backed by forum threads, blog posts, and GitHub issues from the community that lived through them.

This isn't a hit piece. It's a technical autopsy. And if you're evaluating Frappe right now, you deserve to read it before you commit.

## The ORM That Stitches Strings and Hopes for the Best

Every web framework lives or dies by its database abstraction. Frappe's ORM was, for most of its history, built on **string concatenation**.

The framework's own engineering blog [puts it plainly](https://frappe.io/blog/engineering/evolving-frappes-orm-for-security-and-flexibility): the central flaw was "validating and sanitising bad inputs instead of explicitly allowing known good inputs." The internal implementation "used strings to stitch together the final query." This isn't a community complaint — it's a confession from the maintainers.

The consequences were predictable. Multiple SQL injection vulnerabilities were discovered and patched **reactively** over the years. Each fix addressed a specific injection vector rather than solving the systemic problem. If your framework's security model is "we'll patch the holes as attackers find them," you don't have a security model — you have a prayer.

### The N+1 Problem Nobody Escapes

Like any ORM, Frappe suffers from the N+1 query problem. But unlike mature ORMs (SQLAlchemy, Django's ORM, ActiveRecord) that provide eager loading, `select_related`, or `prefetch_related` out of the box, Frappe's answer was... write a JOIN yourself.

The typical Frappe anti-pattern looks like this:

```python
# Fetch all sales orders
orders = frappe.get_all("Sales Order", fields=["name", "customer"])

# Now fetch items for EACH order — one query per order
for order in orders:
    order["items"] = frappe.get_all(
        "Sales Order Item",
        filters={"parent": order["name"]},
        fields=["item_code", "qty", "rate"]
    )
```

With 500 orders, that's 501 database queries. The "fix" is to write a raw SQL JOIN — which defeats the purpose of having an ORM. The framework's own documentation acknowledges this and recommends the JOIN approach, but that leads directly to the next problem.

### The 97% Trap

Frappe's maintainers estimate that **97% of data manipulation can be done through the ORM's building blocks**. That sounds great until you realize the remaining 3% — the complex reports, the multi-table aggregations, the queries that actually matter in a business application — require you to "bypass the ORM layer and write raw SQL queries yourself."

And when you bypass the ORM, you also bypass Frappe's permission model. The [engineering blog](https://frappe.io/blog/engineering/evolving-frappes-orm-for-security-and-flexibility) admits this means developers must "reimplement Frappe's non-trivial permission model to avoid leaking unauthorised data." So you're writing raw SQL *and* reimplementing authorization. At that point, what is the framework giving you?

A community audit found approximately **694 uses of `frappe.db.sql()` across 263 files** in the codebase — raw SQL scattered throughout what's marketed as a "low-code" framework. That's not an escape hatch being used in edge cases. That's load-bearing raw SQL.

<Callout type="info">
**Version 16 update:** Frappe has introduced a unified query builder based on pypika. It's a genuine improvement. But it doesn't retroactively fix the years of raw SQL already embedded in ERPNext and every custom Frappe app ever written.
</Callout>

The consequences of this history aren't abstract. Frappe has a dedicated [vendor page on CVEDetails](https://www.cvedetails.com/vendor/17053/Frappe.html) with a documented track record of SQL injection CVEs. CVE-2025-56381, for which a public PoC exists, is a recent example of the class of issue that emerges when a framework's ORM was historically built on string stitching. Chained with another vector — an administrator misconfiguration or a second SQLi flipping the right flag — these issues can escalate to wholesale data exposure. That's not a bug. That's a consequence of the original design.

## Child Tables: Where Data Modeling Goes to Die

Frappe's "child table" concept is central to how you model one-to-many relationships — line items in an invoice, rows in a material request, entries in a journal. On the surface, it's elegant. In practice, it's a minefield of artificial constraints.

### No Nesting Allowed

You cannot nest a child table inside another child table. [This has been requested since at least 2017](https://github.com/frappe/frappe/issues/17503), reopened [multiple times](https://github.com/frappe/frappe/issues/23307), and remains unsupported. In the real world, data is hierarchical. A production order has operations; each operation has material inputs. Frappe says: flatten it or hack around it.

### No Independent Controllers

Child tables don't have their own controllers. All validation logic for child rows must be [crammed into the parent document's controller](https://medium.com/@prabhsimar.s.sawhney/the-cost-of-using-child-tables-in-frappe-framework-4e48a7d9c1ca). Validating stock availability for each line item in an order? You're writing a loop inside the parent's `validate` method, mixing parent-level and row-level concerns into a single, increasingly unmanageable function.

### Permission Black Hole

Here's the one that catches people off guard: **non-admin users cannot directly query child tables using Frappe's ORM**. Methods like `frappe.get_all()` and `frappe.db.get_list()` fail for non-privileged users trying to retrieve child table rows. The workaround is to write a whitelisted server method or — you guessed it — raw SQL.

### Display Constraints

The grid UI for child tables is limited to displaying **10 columns** of data. If your line item has more than 10 fields that users need to see simultaneously, you're out of luck. The framework's UI simply won't show them in the grid view.

These aren't edge cases. One-to-many relationships with validation, permissions, and reasonable display requirements are the *bread and butter* of business applications — the exact domain Frappe claims to serve.

## The Permission System That Fights Itself

Frappe's permission model has three layers, each capable of overriding the one beneath it: role-level permissions (`tabDoc Perm`), user-level restrictions (`tabUser Perm`), and document sharing (`tabDoc Share`). On paper that's defense-in-depth. In practice each layer can override the one below it, and the surface area required to reason about access is enormous: DocPerm, DocShare, User Permission, Role Profile, Permission Manager, Module Profile, Custom DocPerm — a parade of DocTypes the framework spawned to hold a permission model together. Configuring access for one user means traversing all of them.

### Allow-by-Default in an Allow-by-Default-is-Dead World

The industry has been moving toward least-privilege for a decade. Zero-trust, deny-by-default, explicit grants — the consensus is that systems should refuse access unless a rule grants it. Frappe runs the opposite playbook: **absence of any role deactivates the permission barrier entirely**. No roles assigned? You're implicitly approved. The moment you add a role, the barrier engages — but only as a flat allow-list of explicit grants. Restriction is a workaround layered on top of an allow-by-default base, not a primitive.

That choice cascades. Every customer-facing app must enumerate every resource a user *should* see, because the default is "see everything in any DocType your role doesn't explicitly deny." Forget one DocType in your restriction list and it leaks. Add a new DocType to the system and existing users implicitly gain visibility unless someone remembered to update the deny rules.

### The Share Bypass

The `tabDoc Share` layer is described in Frappe's own documentation as a permission override — and it lives up to the name. A user with share permission on a document can share it with anyone and **grant them write access**, regardless of what role-level or user-level permissions say. The docs put it plainly: it "bypasses every check mentioned above."

The implication: a user with read-only access who also has share permission can effectively promote any other user to write access on that document. That's not an override — that's a privilege escalation vector baked into the framework.

### Permissive-by-Default

Frappe Cloud's own documentation admits: *"Implementation of access controls is not complete. There maybe hiccups here and there."*

That quote is from Frappe's official documentation for their own cloud product. When the people who built the system feel the need to disclaim their access controls in the docs, the signal is clear.

The user-level restriction system (`tabUser Perm`) operates on a permissive-by-default model: absence of any role results in deactivation of the permission barrier, meaning users with no roles assigned get implicit approval on resources not explicitly denied. The moment you create a role and add nothing to it, the barrier activates — but "all permissions must be explicit" means a team of 10 organizations, each with 6 resource types, requires 60 explicit permission records per user just to baseline the system.

### The Scale Cliff

This isn't theoretical. In one production multi-tenant setup we analyzed, the system had accumulated **28,272 User Permission override documents** — just for permission management. A single admin-level user required **2,010 individual permission overrides** (335 organizations × 6 resource types per org). For a flat org structure, that growth is linear. Add a second tier — a "grandfather" org above multiple "father" orgs — and it becomes multiplicative.

At ~40 organizations, permission sync operations begin timing out. UX degrades. The system needs a manual "Resync" button as a workaround. The workaround then breaks at scale, requiring another workaround.

There's a second, smaller cliff hardcoded directly into Press: permission queries fetch documents with `limit=9999`. If a team has 10,000+ sites, the query silently truncates and the permission check operates on an incomplete picture. Loading 9,999 documents into memory just to filter for access is also, by any reasonable definition, not a permission system — it's a brute-force scan with a magic number for a guardrail.

### Five Permission Systems, None of Them Talking

When Frappe builds products on top of itself, the permission sprawl compounds. Frappe Press (Frappe's own cloud hosting platform) ended up with **five separate permission mechanisms** layered on each other:

1. `hooks.py` `has_permission` / `permission_query_conditions` — core Frappe hooks
2. `@protected` decorator — described in their own codebase docstring as "stupid" and "magical," tries four different heuristics to guess which document you're accessing
3. `@role_guard` decorators — granular role checks on API endpoints only
4. `action_guard` decorator — action-specific permissions
5. Press Permission Groups / Press User Permissions — method-level JSON config

These don't integrate. If a user accesses documents through standard Frappe APIs instead of Press's custom APIs, the granular permissions (layers 3-5) are bypassed entirely. A security auditor trying to answer "can User X do Action Y on Document Z?" would need to trace through all five systems. The codebase documents a `skip_roles()` hack with comments reading `# FIX: Not a good solution. Needs to be rethought` and `# HACK: Skips all validation if the user has no roles assigned`. No roles assigned = all role-based permission checks skipped. It's backwards.

Frappe Press also completely ignores Frappe's built-in `DocShare` feature — a search for `frappe.share` in the Press codebase returns zero results. They built an entirely separate permission system from scratch because the built-in one couldn't serve their use case. The framework's signature feature, replicated from the ground up, inside the framework's own flagship product.

## The Monolith That Ate Everything

Frappe's architecture is monolithic by design and by philosophy. That's a valid choice — monoliths can be excellent. But Frappe's monolith isn't the "well-structured modular monolith" kind. It's the "everything depends on everything" kind.

A [community discussion on microservices vs. monolith](https://discuss.frappe.io/t/microservice-vs-monolith-mindset/74569) laid the coupling bare. ERPNext bundles Agriculture, Healthcare, Education, Manufacturing, Retail, and more into a single application where modules know each other's internals. The inheritance chain tells the story: Purchase Orders roll up into Buying Controller, which rolls up into Stock Controller. These aren't clean interfaces — they're behavioral assumptions baked into a class hierarchy. Want purchasing without stock management? You're fighting the framework.

The deployment unit makes this worse. Deploying a Frappe app means deploying an entire **bench** — Frappe core, your app, Redis, the background worker, and web server configuration, all glued together. There's no lightweight microservice deployment. One app, one bench, one full stack. Five small services means five benches, five Redis instances, five sets of background workers.

## Memory Management by Brute Force

Frappe uses `gc.freeze()` — Python's garbage collector freeze — to reduce memory usage in its Gunicorn workers. The idea is sound: freeze all pre-loaded objects into a "permanent generation" that the GC ignores, so forked worker processes can share that memory via Copy-on-Write rather than duplicating it. Frappe claims ~50MB+ savings per worker.

The cost isn't in the savings. It's in what happens after.

### Objects That Live Forever

`gc.freeze()` doesn't pause the GC. It permanently exempts pre-fork objects from collection. Any object created *after* the freeze that holds a circular reference back to something frozen — an imported module, a cached DocType, a global — becomes immortal. The GC will never collect it.

RQ background workers fork constantly. Each fork inherits the frozen objects from the parent. Each job creates new objects. Some of those objects have circular references to the frozen layer. They accumulate. The worker slowly bloats. Frappe's solution: `max_jobs`, which restarts workers after a fixed number of jobs to reclaim memory. Their own codebase comments describe this as a band-aid.

### The Debugging Blind Spot

When you freeze the GC, standard debugging tools stop working:

- `gc.get_objects()` won't return frozen objects
- Memory profilers like `objgraph` give incomplete pictures
- `gc.get_referrers()` and `gc.get_referents()` miss frozen object relationships

You won't see exceptions. You'll see gradual memory growth, OOM kills, workers dying mysteriously. The framework self-reports this: their codebase tracks `LAST_MEASURED_USAGE = 46  # MB for a simple background job` with a comment that it "creeps up" and the "potential root cause can be a newly added python module import." Every new import permanently bloats the frozen generation.

### State Pollution Between Jobs

`frappe.local` is thread-local storage holding the database connection, session, user, flags, cache, and dozens of other globals. In non-forking workers (`FrappeWorkerNoFork`), jobs run in the same process — meaning the same `frappe.local` survives across jobs. Frappe's `destroy()` calls `release_local()` to clear it, but that only clears the thread-local dict. Module-level caches decorated with `@lru_cache` and `@site_cache` are not cleared. Global dicts like `frappe.controllers`, `frappe.whitelisted`, and `frappe.loggers` persist. Frozen objects may still hold references to old `frappe.local` state.

The result is uncomfortable: Job 2 may see stale data from Job 1. A polling job processing the wrong site's context. A billing job reading frozen plan data. Users seeing other users' sessions. Permission checks running against the wrong user. None of these throw errors — they corrupt outputs silently.

### The Smoking Gun in Press's Codebase

The fingerprint of these issues is visible in Frappe Cloud's own code. Press ships scheduled functions named `fail_stuck_press_jobs`, `fail_old_jobs`, and `mark_stuck_updates_as_fatal`. There is a function whose entire purpose is to fail jobs that got stuck. Step timeouts are hardcoded to **18000 seconds** (5 hours) for individual migration steps — long enough that even a memory-leaking worker will probably finish before being killed. These aren't features. They're admissions in code form.

## The WSGI Tax

Frappe runs on WSGI — one thread per request, no async I/O. Profiling Frappe requests with `py-spy` reveals that the actual user code handling accounts for only **53.63% of request time**. Framework initialization takes another **23.86%**. Response serialization takes **22.49%**. That's a **~46% framework overhead** on every request, before your code runs.

ASGI — the async successor specification adopted by Django, FastAPI, and Starlette — handles concurrent connections via async I/O instead of thread-per-request. Frappe cannot adopt it. Its ORM, database layer, and numerous dependencies are synchronous-only. Even if you deployed Frappe behind an ASGI server, the framework would weigh it down and make it slower than pure WSGI. The path to async is permanently blocked by architectural decisions baked in at the foundation.

## The Worker That Can't Stand Alone

Frappe's background workers — the RQ-based job runners — are not standalone Python processes. They are **Frappe processes**. Every worker boots the entire Frappe runtime: load `frappe.controllers`, hydrate every DocType meta, connect to MariaDB and Redis, register hooks, populate `frappe.local`. Then, after all that, it pulls a job off the queue.

This means a worker doing nothing — idle, waiting for a job — sits at hundreds of MB of resident memory. Spawn ten workers and you've spent gigabytes before processing a single job. Want a lightweight job runner that just sends an email or hits an API? Doesn't exist. Every job pays the framework tax. Every worker is a fully booted Frappe instance whether it needs to be or not.

The coupling goes deeper than memory. Job code can't import from "just Frappe utilities" — it imports from `frappe`, which means `frappe.init()` must have run, which means a site context must exist, which means MariaDB and Redis must be reachable. A unit test for a job function can't run without a full bench. CI pipelines pay this tax on every commit. New team members pay it on day one — a forum thread titled [*"Why this framework is so frustrating to work with?"*](https://discuss.frappe.io/t/why-this-framework-is-so-frustrating-to-work-with-always-unexpected-things-keep-happening/128494) opens with the developer noting "it takes an immense amount of effort to get the bench utility set up on your local system." Background schedulers keep running for inactive sites, generating exceptions that cause broken pipe errors in *active* sites. Services persist on ports after abrupt terminal closures. The runtime doesn't just have rough edges — it has hidden trapdoors.

### RQ's "Horse" Errors and Mystery Crashes

The RQ library Frappe uses forks a "work horse" subprocess for each job. When that horse dies unexpectedly — OOM, segfault, signal — Frappe surfaces it as a generic `Work-horse terminated unexpectedly; waitpid returned ...` error. No stack trace from the job itself. No indication of *why* the horse died. The user-visible job status is "failed," but the actual cause lives in `dmesg`, kernel logs, or nowhere.

Combine this with the `gc.freeze()` memory leaks, the `frappe.local` state pollution, and the lack of structured error reporting, and debugging a flaky background job becomes archaeology. Forum threads on horse errors loop through the same advice (raise memory limits, restart workers, increase timeouts) because nobody can pinpoint a single root cause.

## Locked Into Redis and MariaDB

Frappe's job queue is RQ, which sits on Redis. Redis is also the cache layer, the session store, the SocketIO backplane, and the pub/sub bus. Lose Redis and Frappe doesn't degrade gracefully — it falls over. There is no pluggable queue backend. Want SQS, RabbitMQ, NATS, or a database-backed queue with transactional outbox semantics? Not supported. Redis is hardcoded into the architecture, and a queue backed by an in-memory store means a Redis crash or eviction loses jobs that were enqueued but not yet picked up.

The database story is worse. Frappe officially supports MariaDB. Postgres support exists but is **second-class and full of holes** — community threads catalogue migration failures, missing features, and DocTypes that ship MariaDB-specific SQL. The framework's own raw `frappe.db.sql()` calls (recall the 694 instances across 263 files) frequently use MariaDB syntax — backtick identifiers, `INSERT ... ON DUPLICATE KEY UPDATE`, MariaDB-specific JSON functions — that doesn't translate to Postgres. The pypika query builder helps for new code; it doesn't fix the legacy.

The implication: if your team has Postgres expertise, Postgres tooling, Postgres-based observability, or a compliance requirement that mandates Postgres, Frappe is not a real option. You're picking the framework *and* its database, and the database choice is MariaDB whether you wanted it or not. For a stack billing itself as a flexible foundation, that's a hard ceiling on architectural choice — set in stone before you write your first DocType.

## Migrations Without a Down Path

Mature web frameworks treat database migrations as a first-class concern. Django's `migrate` produces `0042_add_status_field.py` files with `forward()` and `backward()` operations. Rails has `db:migrate` and `db:rollback`. Sqitch, Flyway, and Liquibase all enforce the basic invariant: every schema change has an explicit up and down path, named, ordered, reviewable.

Frappe's `bench migrate` does not. There is no `*-up.sql` / `*-down.sql` pair. Schema changes happen as a side effect of DocType JSON edits — modify a DocType definition, run migrate, and Frappe diffs the JSON against the live schema and applies whatever ALTER statements it thinks are needed. There is no rollback. If a migration breaks, your options are: fix forward, restore from backup, or surgically reverse the schema by hand. There is no `bench migrate --rollback 1`.

Add custom patches via `patches.txt`, which lists Python files run once per site. Each patch is imperative code — no down version, no checksum, no enforced ordering beyond append-order. The framework's response to "how do we evolve the schema safely?" is "be careful and have backups." For an ERP that businesses run their accounting on, that's an alarmingly thin guarantee.

This also rules out **rolling deployments**. A Frappe upgrade requires `bench migrate`, which requires schema changes against a database that can't tolerate two app versions reading it simultaneously. Production deploys are stop-the-world: drain traffic, run migrate, restart bench, hope. Zero-downtime deployment — table-stakes for any service-grade web app in 2026 — is structurally not on the menu.

It's worse than just "restart the bench." Every release flips the site into **maintenance mode** — `bench --site <site> set-maintenance-mode on` — which serves a holding page to every user while the migration runs. The framework hardcodes this into its own update flow (`bench update` toggles it on, runs migrate, toggles it off). For a site with active users, that's a visible outage on every release. For a multi-tenant Frappe Cloud setup with hundreds of sites, it's an outage propagated across every tenant in sequence. There is no "drain one node, migrate, swap." There is only "everyone stops, the database changes, everyone resumes."

## The Administrator User Horror

Every Frappe site has a special `Administrator` user. It is not a role. It is a user account, hardcoded throughout the framework, that bypasses every permission check. Most CLI operations run as Administrator. Many internal API calls assume Administrator context. Migrations execute as Administrator. Background jobs frequently run as Administrator because nothing else has the rights to do what they need.

The consequences:

- **No real least-privilege model for system code.** If a custom app needs to update a record outside the calling user's scope, the path of least resistance is to elevate to Administrator (`frappe.set_user("Administrator")`). This pattern is everywhere in ERPNext and community apps.
- **Audit trails lie.** Modifications made via background jobs or system scripts show "Administrator" as the modifier — useless for forensic analysis.
- **One foot-gun, one keyhole.** If the Administrator account is ever compromised — leaked session, weak password reset flow, SQLi pivot — there is no defense in depth. Administrator can do anything to anything.
- **Testing is contaminated.** Because so much code assumes Administrator context, writing tests that exercise non-admin behavior requires explicit user switching, and any code path that forgets to switch silently runs with full privileges.

The framework was built around a god account, and the god account leaks into every layer.

## The Developer Experience: Python Without the Benefits

In 2026, modern Python is a strongly-typed, statically-checkable language. Type hints are universal. Pydantic, mypy, and pyright catch a class of bugs at edit-time that used to require runtime tests. FastAPI uses type hints to derive request schemas. SQLModel uses them to derive table definitions. The standard now is: if it isn't typed, it isn't reviewable.

Frappe's codebase is largely untyped. DocType controllers receive `self` of type `Document` — a generic bag where every field is `Any`. Whitelisted methods take untyped `**kwargs` from HTTP. `frappe.get_doc()` returns `Document`, regardless of which DocType you asked for. There's no `SalesOrder` class with typed fields you can autocomplete; there's a runtime-resolved blob you guess your way through.

What this means in practice:

- IDE autocomplete is useless on DocType fields. You memorize them or grep the JSON.
- Refactoring is dangerous. Renaming a field requires a project-wide string search, not a type-aware rename.
- Bugs that mypy would catch in five seconds — typo in a field name, wrong argument order, returning `None` where a string is expected — surface only at runtime, often in production, often inside a background job that fails silently.
- Library APIs are inconsistent. `frappe.db.get_value()` returns `Any | None | tuple | dict` depending on arguments. `frappe.get_all()` returns a list of dicts unless you pass `as_dict=False`. The same function mutates its return shape based on flags.

The framework predates the typing era and has not retrofitted. Newer Frappe code adds annotations sporadically, but the core APIs and the auto-generated DocType layer remain untyped. You write Python in 2026 with the developer experience of Python in 2014.

## The Notification DocType: Markdown, Jinja, and Error Logs in a Trench Coat

Frappe ships a "Notification" DocType — a UI for configuring email, SMS, and system notifications triggered by document events. The pitch is low-code: pick a DocType, pick an event, write a template, save. The reality is a fragile pipeline of three layered template engines stitched together, and any of them can fail silently.

The template body is Markdown. The Markdown is rendered with Jinja for variable interpolation. The Jinja runs inside Frappe's sandbox, which exposes a curated subset of `frappe.*` calls. Get a single brace wrong, reference a field that doesn't exist on the bound document, or call a server method that hits a permission edge — and the notification doesn't send. Worse, the failure isn't surfaced to the user who configured it. It lands in the **Error Log** DocType as a stack trace, where it sits until someone notices the error log table has grown to a million rows.

The compounding effect: notifications that worked yesterday silently stop working today because a custom field got renamed, a hook changed the document's shape, or an upgrade altered the sandbox's allow-list. The user thinks the system is fine. The accountants stop getting their alerts. The error log piles up. The fix is invariably "go to /app/error-log and read the trace yourself" — a workflow that assumes every business user is also a Python developer.

## A Frontend Frozen in 2015

In 2026, Frappe's desk interface — the primary UI that users and developers interact with — is built on **jQuery and Bootstrap**.

Let that sink in. The JavaScript ecosystem has gone through Angular, React, Vue, Svelte, Solid, and back again. Component-based architecture, virtual DOMs, reactive state management, TypeScript, server components — a decade of frontend innovation. Frappe's desk is still manipulating the DOM with `$('.some-class').toggle()`.

The team *knows* this is a problem. They've built [Frappe UI](https://github.com/frappe/frappe-ui), a separate component library based on Vue.js and TailwindCSS, used in newer products like Gameplan and Frappe Cloud's dashboard. But the core framework — the one you interact with when building DocTypes, writing client scripts, customizing forms — remains jQuery.

This creates a split-brain developer experience. New Frappe products use modern tooling. The framework itself does not. Client scripts are written in a jQuery-flavored API (`frm.set_value`, `cur_frm.refresh_fields`) that feels like writing code for Internet Explorer 9. If you've hired frontend developers in the last five years, they've never written jQuery professionally and shouldn't have to start now.

## Upgrade Roulette: Every Major Version Is a Coin Flip

Version upgrades in Frappe/ERPNext are not migrations — they're expeditions. The forum is littered with battle reports:

- **v12 to v13**: One developer [reported taking 8 attempts and an entire day](https://discuss.frappe.io/t/after-painful-upgrade-v-12-v-13-error-upon-login/75378), repeatedly hitting "uncommitted changes in frappe app" errors. Another encountered [bench migrate failures](https://discuss.frappe.io/t/bench-migrate-issue-during-the-process-of-v13-upgrade-from-v12/75201) during the switch-to-branch process.

- **v13 to v14**: Users hit [duplicate key errors in Payment Ledger Entry](https://discuss.frappe.io/t/error-in-migrate-from-version-13-to-version-14/119830) after processing 85,000 records, and [database schema issues with missing columns](https://discuss.frappe.io/t/upgrade-issue-database-schema-changes/115531).

- **v15 to v16**: The [upgrade guide](https://discuss.frappe.io/t/guide-to-upgrading-bench-frappe-and-erpnext-from-v15-to-v16-on-debian-12/159062) generated multi-page forum threads of users getting stuck at various stages.

The pattern is consistent: new versions change database structures in ways that custom apps and even stock migrations can't handle cleanly. If you have custom DocTypes, custom fields, or — heaven forbid — direct database modifications, every major upgrade is a roll of the dice.

For an ERP system — software that businesses depend on for accounting, inventory, and payroll — this is not an inconvenience. It's a liability.

## Contributing: Abandon All Hope

Open source lives and dies by its contributor experience. Frappe's is described, by contributors themselves, as ["a total waste of time"](https://discuss.frappe.io/t/contributing-to-frappe-frappe-feels-like-a-waste-of-time/150437).

The complaints are specific and damning:

- **PRs vanish into the void.** Contributors submit pull requests that receive "no feedback or code review" except from automation bots, sometimes for months.
- **No codeowner clarity.** There's "no way of knowing which users are maintainers or employees at Frappe Technologies, versus ordinary community members." Contributors must "read through various issues and PRs, and make wild guesses" about who owns what.
- **Inconsistent timelines.** Some PRs merge in a day. Others take three months with no predictable path to resolution.
- **Reopen cycles.** Contributors may need to reopen PRs multiple times, refactor based on feedback, and repeat indefinitely with no guarantee of acceptance.

When your established developers describe contributing as frightening and wasteful, you don't have a contributor pipeline — you have a contributor graveyard.

## The Escape-Hatch Culture

When the framework's primitives can't express what a developer needs, the codebase reaches for escape hatches. Three show up everywhere, and each one corrodes the layer it punches through.

**`skip_roles()`** — when role checks fail, skip them. Press's own codebase carries the function with a TODO to remove it. Production code calls it because the role engine doesn't model the actual access pattern. Every call is a permission check that didn't happen.

**`frappe.db.commit()`** in the middle of business logic — Frappe wraps each request in a transaction that commits at the end. When that's inconvenient (a long job that wants to checkpoint, a controller that wants to side-effect before raising), code calls `frappe.db.commit()` mid-flow. Now you have partial writes that survive a later failure. Atomicity becomes a suggestion. The roll-forward debugging experience is its own special hell: half the data is there, half isn't, and the only way to know which is which is to read the code path.

**`ignore_permissions=True`** — `frappe.get_doc(...).save(ignore_permissions=True)`. Sprinkled through ERPNext and community apps. Means "I know permissions exist, I'm telling them to stay out of my way." Often correct (system-level code shouldn't be subject to user-level checks), often wrong (a hot path quietly bypassing the permission model because somebody hit a bug in 2017 and never looked back).

These aren't anti-patterns the community invented. They're the framework's recommended workarounds when its abstractions don't fit. The cost: every escape hatch is a place where the guarantees you thought you had — auditability, atomicity, authorization — silently don't apply.

## No Middleware Layer

Every modern Python web framework has a middleware concept. Django has middleware classes. Flask has `before_request`/`after_request`. FastAPI has dependencies and middleware. The pattern is universal because cross-cutting concerns — auth, rate limiting, request logging, tracing, tenant resolution, CSRF — belong in one place that runs on every request, regardless of which view handler executes.

Frappe doesn't have one. It has hooks (`before_request`, `after_request`) that are roughly middleware-shaped, but they don't compose, don't short-circuit cleanly, and don't have a documented ordering contract between apps. There's no idiomatic place to plug in OpenTelemetry, no clean way to layer a rate limiter, no standard pattern for tenant-scoped request validation. Teams either monkey-patch `frappe.handler`, fork the routing layer, or stand up a separate proxy in front of the bench to do what middleware would do in any other framework.

The downstream effect: observability is bolted on per-app rather than installed once. Auth additions live in scattered hooks. Cross-cutting changes touch dozens of files. The framework that markets itself as full-stack quietly assumes you won't need the architectural primitive every other full-stack framework treats as table stakes.

## Patches on Patches

Step back and the pattern across all of the above is the same. The ORM was string concatenation, then patched with input sanitization, then patched with a query builder that doesn't cover legacy code. The permission system grew to three layers, then five in Press, then a `skip_roles()` hack with a TODO to remove it. Memory management was patched with `gc.freeze()`, then patched again with `max_jobs` worker restarts, then patched again with `fail_stuck_press_jobs`. Migrations were patched with `patches.txt`. Schema diffing was patched with `bench migrate`. Permission scaling was patched with a "Resync" button.

Nothing in Frappe was redesigned. It was patched. Each patch addresses a specific failure mode without revisiting the assumption that produced it. The result is a framework where every layer has a workaround stapled to it, and every workaround has its own edge cases that someday will get their own workaround.

## The Verdict: Optimized for Demos, Not Production

Frappe is genuinely impressive in a demo. Define a DocType, watch the form appear, click through the auto-generated API, marvel at the permission matrix. In thirty minutes, you have something that looks like it would take weeks to build from scratch.

But demos don't have 500 concurrent users hitting N+1 queries. Demos don't need nested child tables with row-level validation. Demos don't need to survive a major version upgrade with 200 custom fields intact. Demos don't have junior developers trying to understand why `frappe.get_all` silently returns nothing for non-admin users querying child tables.

**The framework optimizes for time-to-first-impression at the expense of time-to-production.**

### Who Might Still Benefit

To be fair — Frappe can work if:

- Your application is simple enough to stay within the 97% ORM coverage
- You don't need deeply nested data models
- Your team is small and can absorb the bench tax
- You're building an internal tool where jQuery-era UI is acceptable
- You won't need to upgrade major versions frequently (or at all)
- You don't plan to contribute upstream

That's a narrow window, and it gets narrower as your application grows.

### What to Consider Instead

If you need a Python full-stack framework: **Django** with Django REST Framework gives you a mature ORM with eager loading, a massive ecosystem, and contributor experience that's the gold standard of open source.

If you need a low-code platform: **Directus**, **Strapi**, or **Refine** offer modern frontends, clean APIs, and architectures designed for the current decade.

If you specifically need an open-source ERP: evaluate with eyes wide open. ERPNext may still be the best *open-source* option in its category — but understand that you're inheriting every architectural decision described above.

### The Real Cost

You found Frappe because it promised to save you time. It will — for the first two weeks. Then you'll spend that time back, with interest, fighting an ORM that stitches strings, child tables that can't nest, upgrades that break schemas, a frontend from 2015, and a contribution process that burns goodwill.

The most expensive software isn't the kind you pay for. It's the kind that's free until you're too deep to leave.
